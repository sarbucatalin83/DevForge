# DevLearn Constitution

## Core Principles

### I. Runtime Independence

The main application MUST have zero runtime dependencies on external APIs or internet
connectivity. The Claude API is used exclusively by the offline content-generation CLI
script (`scripts/generate-content.js`) which is run manually as a separate step.

- The app MUST start and operate fully with only `docker` installed and `npm install && npm run dev`.
- No API key, no network call, and no cloud service MAY be required to run the app or
  evaluate user answers at runtime.
- The app MUST ship with a starter set of pre-generated content so it is immediately usable
  without running the generation script first.

### II. Pluggable Execution Engine

Every language executor MUST be registered through a config file entry — not through
application code. Adding a new language (Go, Rust, etc.) MUST require only: a Docker image
name, a file extension, a run command, and a test command. No application source-code
changes are permitted to support a new language.

- All user-submitted code MUST run exclusively inside Docker containers — never on the host
  process or in any unsandboxed in-process evaluator.
- The executor config schema MUST be documented and stable; breaking changes to it require a
  constitution amendment.

### III. Content as Data

All questions, answers, explanations, and coding exercises MUST be stored as plain JSON files
on disk. They MUST be human-readable, hand-editable, and safe to commit to version control.

- The content JSON schema MUST be language-agnostic from day one so that future language
  tracks (Go, Rust, etc.) use the same format with no structural changes.
- No database (SQL, embedded, or otherwise) MAY be introduced in v1 for content storage.
- Content files generated by the CLI script MUST be the single source of truth; the app MUST
  NOT modify them at runtime.

### IV. Security by Isolation (NON-NEGOTIABLE)

User-submitted code MUST NEVER be executed on the host process. Every execution MUST happen
inside a Docker container.

- Every container MUST enforce: a hard execution timeout (default 10 s, configurable),
  a CPU cap, and a memory cap.
- Containers MUST be removed immediately after execution completes or times out.
- No exception, shortcut, or "dev mode" bypass of container isolation is permitted. This
  principle CANNOT be relaxed for any reason including performance or convenience.
- Any future code path that would execute user-supplied code outside a container is a
  critical security defect and MUST be treated as such.

### V. Self-Assessment Over Automation

Written quiz answers MUST be evaluated by the user, not by an algorithm or API call. After
submission the app MUST reveal the stored correct answer and its explanation side by side
with the user's answer. The user then self-rates with an explicit **I got it** / **I missed
it** action.

- No runtime API call MAY be made to evaluate a user's written answer.
- Correct answers and explanations are generated offline by the CLI script and stored with
  the question; the app MUST surface them exactly as stored.
- A **Show Answer** button MUST be available at any point before submission for users who
  choose to read the answer directly.

### VI. Simplicity Ceiling

Complexity MUST only be added when a feature strictly requires it. YAGNI applies to all
infrastructure and architectural decisions.

- No authentication, no cloud database, no multi-user support, and no server-side session
  state MAY be introduced in v1.
- Progress MUST be stored in `localStorage`; no backend persistence layer is permitted for
  user state in v1.
- Each new abstraction or dependency MUST be justified against a concrete present requirement,
  not a hypothetical future one.

## Technology Constraints

The following choices are locked for v1 and MUST NOT be substituted without a constitution
amendment:

| Layer | Choice |
|-------|--------|
| Frontend framework | React + Vite |
| UI component library | shadcn/ui (Tailwind-based) — no other CSS frameworks |
| In-browser code editor | Monaco Editor |
| Backend runtime | Node.js + Express (local server only) |
| JS/TS test runner (in Docker) | Jest |
| Styling | Tailwind CSS via shadcn only |

Additional constraints:

- No ORM, no SQL database, no Redis MAY be introduced in v1.
- The backend MUST remain a local-only server; no cloud hosting configuration is permitted
  in v1.

---

## Scope Boundary

The following are explicitly **out of scope for v1** and MUST NOT be designed for,
accommodated, or partially implemented:

- Mobile or tablet layout (minimum supported viewport: 1280 px wide)
- User accounts or multi-user support
- Leaderboards, social features, or gamification beyond pass/fail tracking
- Cloud deployment or remote hosting
- AI-graded code evaluation (unit tests are the sole judge of code correctness)
- Paid or subscription content

Any feature request that touches these areas requires a constitution amendment before
work begins.

---

## Governance

This constitution supersedes all feature specifications, implementation plans, and task
lists. Where a lower-level document conflicts with this constitution, the constitution wins.

**Amendment procedure**: Any change to a principle requires (1) explicit documentation of
what changes and why, (2) review against all dependent templates and specs, and (3) a
version bump following the policy below before work begins.

**Non-negotiable principles**: Principles **II (Pluggable Execution Engine)** and
**IV (Security by Isolation)** CANNOT be amended without a full architecture review. Any
pull request that would violate either principle MUST be blocked regardless of other
approvals.

**Versioning policy**:
- **MAJOR**: Backward-incompatible removal or redefinition of a principle.
- **MINOR**: New principle or section added, or material expansion of existing guidance.
- **PATCH**: Clarifications, wording fixes, non-semantic refinements.

**Compliance**: Every implementation plan (plan.md) MUST include a Constitution Check gate
that explicitly verifies compliance with all six principles before Phase 0 research begins
and again after Phase 1 design.
