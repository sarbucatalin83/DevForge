# DevLearn — Local Developer Learning Platform

## Overview

A locally-running web application for learning JavaScript, TypeScript, and React through interactive
tests and coding exercises. Designed to serve all skill levels from absolute beginner to senior
engineer. The architecture is intentionally extensible so that Go, Rust, and other languages can be
added in the future without structural changes.

## Tech Stack

- **Frontend**: React + Vite + shadcn/ui + Monaco Editor (the VSCode engine)
- **Backend**: Node.js / Express — local server, no cloud dependency at runtime
- **Code execution**: Docker containers, one image per language runtime
- **Content generation**: Separate CLI script that uses the Claude API — run once offline to populate
  content files; the main app has no Claude dependency and needs no API key to operate
- **Persistence**: Structured JSON files on disk (questions & exercises) + browser localStorage (user progress)

---

## Feature 1 — Learning Tracks

Three initial tracks: **JavaScript**, **TypeScript**, **React**. Each track is fully independent
and covers every topic from trivial/beginner through advanced/senior level. Content is
AI-generated by Claude, designed to leave no gap in the curriculum.

- Free navigation: users can jump to any topic or difficulty at any time — no forced linear order.
- Sidebar filter: filter exercises by **track**, **topic**, and **difficulty level**
  (Beginner / Intermediate / Advanced / Senior).
- The content schema is language-agnostic so adding a new track (Go, Rust, etc.) requires only
  a new content bundle and a Docker executor config, with zero frontend changes.

---

## Feature 2 — Written Answer Quiz Mode

A flashcard-style question and answer mode using open text only (no multiple choice).

### Content generation
- Claude generates **question + correct answer pairs** for every topic and difficulty level in each track.
- Generated pairs are stored as structured JSON files on disk (pre-generated, not on-the-fly).
- A **Regeneration UI** lets the user select a track, topic, and difficulty and trigger Claude to
  produce a fresh batch of Q&A pairs, replacing or appending to the stored set.

### Quiz flow
1. User reads the question and types a free-text answer in a text area.
2. User clicks **Submit** — the app immediately reveals the stored correct answer side by side with
   the user's answer. The user reads both and clicks **I got it** (pass) or **I missed it** (fail)
   to self-assess. No API call is made.
3. The correct answer and a stored explanation (generated offline with the content script) are always
   shown after every submission, pass or fail, as a learning moment.
4. A **Show Answer** button is available at any point before submitting for when the user is stuck.

## Feature 3 — Code Editor Exercise Mode

An in-browser coding environment powered by the Monaco editor.

### Exercise types (all four must be supported)
- **Fill in the blank**: starter code with gaps marked; user fills in the missing parts.
- **Fix a bug**: broken code is provided; user identifies and fixes the defect.
- **Implement a function**: a function signature and description are given; user writes the body.
- **Refactor**: working but low-quality code is provided; user restructures it to meet a stated goal.

### Content generation
Claude generates each exercise as a bundle stored in a JSON file:
- `description` — plain-language task description shown to the user.
- `starterCode` — the code pre-loaded into the editor.
- `unitTests` — test suite that will be executed against the submission.
- `solution` — the reference correct solution, hidden until after evaluation.

### Exercise flow
1. User reads the task description and edits code in the Monaco editor.
2. User clicks **Submit**.
3. The backend sends the user's code plus the unit test suite to a Docker container for the
   relevant language runtime.
4. The container executes the tests and returns results.
5. **Pass** (all tests green): show a success state with test output.
6. **Fail** (one or more tests red): show which tests failed and reveal the correct solution side
   by side with the user's code.
7. No hints — the only feedback path is submit → result → optional solution reveal.

## Feature 4 — Docker Execution Engine

- Each language has a dedicated, minimal Docker image containing only the language runtime and a
  test runner.
- The Node.js backend spawns a container on demand per submission, mounts the user code and test
  files, runs the tests, captures stdout/stderr, then removes the container.
- Containers are **time-limited** (configurable timeout, default 10 s) and
  **resource-constrained** (CPU + memory caps) to prevent runaway processes.
- The executor is pluggable: a language is registered by adding an entry to an executors config
  file that specifies the Docker image, file extension, run command, and test command. No code
  changes required to add a new language.
- Initial executor: **Node.js** with **Jest** as the test runner.
- Planned executors (same pluggable pattern): Go (with `go test`), Rust (with `cargo test`).

## Feature 5 — Content Generation (Separate CLI Script)

Content generation is fully decoupled from the main app. It is a standalone Node.js CLI script
(`scripts/generate-content.js`) that requires an `ANTHROPIC_API_KEY` and is run independently,
before or alongside the app, to populate the local content files.

### CLI usage
```
node scripts/generate-content.js \
  --track javascript \
  --topic closures \
  --difficulty intermediate \
  --type quiz \
  --count 20
```

### What the script does
- Calls Claude with a structured prompt to generate Q&A pairs or coding exercise bundles.
- Each Q&A pair includes: `question`, `answer`, `explanation`.
- Each coding exercise bundle includes: `description`, `starterCode`, `unitTests`, `solution`, `type`
  (fill-blank | fix-bug | implement | refactor).
- Output is written to `content/<track>/<topic>/<difficulty>-<type>-<timestamp>.json`.
- Re-running appends a new timestamped file; an optional `--replace` flag overwrites the latest.

### Content browser (in-app, read-only)
- A **Content** page inside the app lists all generated files grouped by track and topic, showing
  item counts per difficulty and type so coverage gaps are visible at a glance.
- No generation is triggered from this page — it is purely informational.

---

## Feature 6 — Progress Tracking

- Stored entirely in **localStorage** — no backend database, no user accounts required.
- Tracked per item: answered (yes/no), last verdict (pass/fail), attempt count, last attempted date.
- Progress summary displayed per track and per topic: e.g. "React → Hooks: 8/20 exercises
  completed, 6 passed".
- A **Reset Progress** option per track or globally.

## UI / UX Requirements

- **Component library**: shadcn/ui throughout.
- **Dark / light mode** toggle, defaulting to system preference.
- **Layout**:
  - Quiz mode: centered single-column card layout.
  - Code exercise mode: two-panel layout — left panel shows task description, right panel is the
    Monaco editor; results appear below or in a drawer.
- **Navigation**: top-level sidebar or tab bar to switch between tracks; secondary filter panel
  for topic and difficulty within a track.
- **Responsive**: functional on laptop screens minimum (1280 px+); mobile is not a priority.
- **Error states**: if Docker is not running or the Claude API key is missing, show clear inline
  error messages with setup instructions rather than generic errors.

---

## Non-Functional Requirements

- Runs entirely on localhost with **no internet dependency and no API key at runtime**.
- The main app setup requires only: `docker` installed and `npm install && npm run dev`.
- Content generation is a separate step: requires `ANTHROPIC_API_KEY` in a `.env` file and running
  the CLI script manually. The app ships with a starter set of pre-generated content so it is
  immediately usable without running the script first.
- Content JSON files are human-readable and can be hand-edited or committed to version control.
- The app must start in under 5 seconds after `npm run dev` on a modern laptop.

---

## Out of Scope (v1)

- User accounts or multi-user support.
- Cloud deployment.
- Leaderboards or social features.
- AI-evaluated code (unit tests are the sole judge of correctness).
- Mobile / tablet layout.
- Paid or subscription content.
